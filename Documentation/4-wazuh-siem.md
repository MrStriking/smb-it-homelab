## Wazuh SIEM


## Overview

Wazuh is an open source SIEM providing features such as centralised log management, threat detection, file integrity monitoring, security configuration assessment, and vulnerability detection across all machines in the lab. Wazuh 4.9.2 was deployed on the Ubuntu Server via Docker. Wazuh agents were installed and setup on the virtual machines in order to send logs to Wazuh. Wazuh agent isn't supported on pfSense but I configured it to send logs to Wazuh via syslog.


## Deployment

Wazuh was deployed using the official wazuh-docker single-node compose file, consisting of three containers:

| Container | Role |
|---|---|
| wazuh.manager | Receives agent logs and generates alerts |
| wazuh.indexer | Stores alert data |
| wazuh.dashboard | Web UI for visualisation and investigation |

Dashboard accessible at: `https://192.168.10.20`

## Features Configured

### File Integrity Monitoring (FIM)
Configured on DC01 to monitor:
- `C:\Users` — user profile changes
- `C:\Windows\System32` — system file changes
- `C:\Program Files` — application changes

FIM alerts are generated in real time when files are created, modified, or deleted in monitored directories.

SCA scans run against the CIS Microsoft Windows Server 2022 Benchmark on DC01. The policy file was modified to support Windows Server 2025, however the checks are still according to the 2022 benchmark making some failed tests unsolvable and the final score not a true value reflecting the security of the server. The score was initially 35%, and after checking the failed tests and applying the recommended remediations in Group Policies, the score went up to 72%.

Notable remediations applied:
- Password policy strengthened to meet CIS requirements
- Account lockout policy configured
- Audit policies enabled
- Various registry hardening settings applied

### Vulnerability Detection

<img width="1666" height="736" alt="image" src="https://github.com/user-attachments/assets/8e92e876-434d-4cb4-a8ad-b8e225cd9e2b" />

Most vulnerabilities detected in Linux kernel packages and Python dependencies on the Ubuntu server. There are a lot of high severity vulnerabilities, however most cannot be addressed from our side due to them being container dependencies related, basically coming from Wazuh and Peppermint which would require fixes from the vendors' sides. The linux kernel would also require testing to avoid breaking dependent services. In a real life scenario these would be tracked in a vulnerability register, prioritised by CVSS score and exploitability, and scheduled for remediation through a formal patch management process.
